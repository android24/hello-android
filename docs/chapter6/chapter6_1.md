# 6.1 为什么 Android 需要异步编程

前面几章里，我们已经让课程 App 具备了页面、跳转、数据存储和架构边界。现在要面对一个更真实的问题：很多事情不是立刻发生的。

网络请求需要等待，数据库读写需要时间，文件处理可能很慢，后台同步不一定马上执行。用户点击按钮后，App 不能僵住；网络失败时，页面也不能像断了线的风筝一样失去方向。

第 6 章要解决的，就是 Android 工程里最常见的一类问题：异步。

## 本章通关画面

完成第 6 章后，你应该能把“刷新课程列表”这件事讲成一条清楚的数据流：

```text
用户点击刷新
  -> ViewModel 启动协程
  -> Repository 请求网络或读取缓存
  -> Flow 持续发出状态
  -> Compose 根据 StateFlow 更新页面
  -> 出错时给出提示
  -> 页面关闭时自动取消无意义任务
  -> 必要时交给 WorkManager 做可靠后台同步
```

这不是为了背协程 API，而是为了让 App 在真实世界里更稳：慢一点没关系，失败了也有路，用户离开页面时不做无用功。

## 第一部分：主线程很忙

Android 主线程负责很多关键工作：

- 绘制界面。
- 响应点击。
- 处理生命周期回调。
- 分发输入事件。
- 更新 Compose 状态。

如果你在主线程里做耗时任务，例如网络请求、复杂计算或大量数据库操作，页面就会卡住。卡住时间过长，用户会感觉 App 不顺滑；更严重时，系统会触发 ANR。

所以 Android 开发里有一条很重要的底线：

```text
主线程负责快事情，耗时任务交给后台。
```

## 第二部分：进程与线程

在进入协程之前，需要先理解两个老朋友：进程和线程。

可以先这样看：

```text
进程：App 运行时拥有的一块独立空间
线程：进程里真正执行代码的工作单元
```

一个 Android App 启动后，通常运行在自己的进程里。这个进程里会有主线程，也可以创建其他后台线程。

主线程负责 UI 和交互，后台线程负责耗时任务。它们同在一个进程里，但分工不同。

如果把 App 进程想成一家小店：

- 进程是这家店的房子和资源。
- 主线程是前台接待，负责面对用户。
- 后台线程是后厨和仓库，负责慢一些、重一些的工作。

前台接待如果跑去后厨熬汤，顾客就没人招呼了。Android 主线程被耗时任务卡住，也是同样的道理。

## 第三部分：Thread 与线程池解决过什么问题

在 Java 时代，我们经常直接使用 `Thread`：

```kotlin
Thread {
    // 执行耗时任务
}.start()
```

这能把任务放到后台，但问题也明显：

- 线程创建和销毁有成本。
- 线程太多会抢资源。
- 结果切回主线程需要额外处理。
- 任务取消和生命周期绑定都不方便。

于是又有了线程池：

```kotlin
val executor = Executors.newFixedThreadPool(4)
executor.execute {
    // 执行后台任务
}
```

线程池的价值是复用线程，避免每个任务都新建线程。它比直接创建 `Thread` 更适合多个任务并发执行。

但在线程池模型下，开发者仍然需要关心很多细节：任务归属、取消、异常、线程切换、生命周期泄漏。

这正是 Kotlin Coroutines 想改善的地方。

## 第四部分：从 Java 线程模型到 Kotlin 协程

协程不是完全取代线程。更准确地说：

```text
线程是系统执行代码的真实载体。
协程是运行在线程之上的轻量任务模型。
```

协程的便捷性主要体现在：

- 可以用顺序代码表达异步逻辑。
- 可以轻松挂起和恢复。
- 可以通过 Scope 绑定生命周期。
- 可以通过 Dispatcher 表达任务适合在哪类线程上执行。
- 可以用结构化并发管理子任务。
- 可以更自然地处理取消。

例如 Java 风格可能会写出“发起任务、回调结果、切回主线程”的嵌套结构；协程则可以写成：

```kotlin
viewModelScope.launch {
    val lessons = repository.refreshLessons()
    uiState.value = uiState.value.copy(lessons = lessons)
}
```

代码看起来是顺序的，但等待期间不会阻塞主线程。

所以第 6 章不是跳过 Java 线程模型，而是先理解它，再理解 Kotlin 为什么把异步写法变得更适合现代 Android。

## 第五部分：异步任务并不神秘

所谓异步，可以先理解为：

```text
我先发起一件需要等待的事情，
等待期间不阻塞页面，
事情完成后再回来更新状态。
```

例如刷新课程列表：

```text
点击刷新
  -> 显示加载中
  -> 后台请求课程数据
  -> 成功后展示列表
  -> 失败后展示错误
```

这和第 2 章的状态思想完全一致：页面不是直接“等结果”，而是根据状态变化重新渲染。

## 第六部分：协程解决什么

Kotlin Coroutines 让异步代码可以写得接近同步代码。

你可以写：

```kotlin
viewModelScope.launch {
    val lessons = repository.refreshLessons()
    uiState.value = LessonUiState.Success(lessons)
}
```

这段代码看起来是顺着写的，但不会阻塞主线程。协程帮你管理等待、恢复、取消和调度。

协程适合处理：

- 网络请求。
- 数据库读写。
- 文件读写。
- 一次性的后台计算。
- ViewModel 中的页面任务。

## 第七部分：Flow 解决什么

协程适合表达“一次任务”，Flow 更适合表达“会不断变化的数据”。

例如：

- Room 表数据变化。
- DataStore 配置变化。
- 搜索输入变化。
- 网络轮询状态变化。
- 页面状态持续更新。

Flow 可以理解成一条会不断流出数据的管道：

```text
数据源 -> map/filter/catch -> UI State -> Compose 页面
```

第 4 章已经接触过 DataStore 和 Room 的 Flow，第 6 章会把它们讲清楚。

## 第八部分：WorkManager 解决什么

协程和 Flow 通常跟随页面或应用进程运行。但有些任务需要更可靠：

- App 退出后仍希望稍后同步。
- 网络恢复后再上传日志。
- 设备充电时执行大任务。
- 每隔一段时间同步课程进度。

这类任务更适合 WorkManager。

可以先这样区分：

```text
协程：当前代码里的异步执行
Flow：持续变化的数据流
WorkManager：系统托管的可靠后台任务
```
