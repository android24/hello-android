# 6.3 结构化并发、取消与超时

会启动协程之后，下一个问题是：这些协程什么时候结束？谁负责取消？失败时会不会拖垮其他任务？

这就是结构化并发要解决的问题。

好的异步代码不只是“能跑”，还要知道任务属于谁、何时取消、失败如何传播、等待多久算太久。

想象一个很常见的场景：用户在课程列表页连续点了三次刷新，然后立刻退出页面。如果代码只是不断启动新任务，旧请求可能还在跑，旧页面可能还在等回调，新的结果和旧的结果也可能抢着更新同一个状态。

这一节就是给这些异步任务“登记户口”：谁发起的，归谁管，什么时候该停，等太久要不要放弃。

## 本节定位

本节继续讲协程基础，但重点从“如何启动”转向“如何管理”。

你会开始理解为什么 Android 推荐 `viewModelScope`，为什么不要随意创建不受管理的全局协程，以及为什么取消不是异常情况，而是正常流程。

## 学习目标

学完本节后，你应该能够：

- 理解结构化并发的基本思想。
- 知道取消在协程中是正常机制。
- 使用 `withTimeout` 或 `withTimeoutOrNull` 控制等待时间。
- 理解父协程和子协程的关系。
- 避免在 Android 中滥用全局协程。

## 第一部分：什么是结构化并发

结构化并发可以先理解为：

```text
任务不是散落在空气里，
而是挂在明确的作用域下面。
父任务结束，子任务也应该结束。
```

例如 ViewModel 启动刷新：

```text
LessonViewModel
  -> viewModelScope
      -> refreshLessons coroutine
          -> request network
          -> read database
```

当 ViewModel 被清理时，刷新任务就没有继续存在的意义，应该自动取消。

## 第二部分：为什么不要随便用 GlobalScope

`GlobalScope` 启动的协程不跟随页面或业务生命周期。

这意味着：

- 页面关闭了，任务可能还在跑。
- 任务失败了，没人知道。
- 多次进入页面，可能留下多个旧任务。
- 很难测试和排查。

在课程里，我们优先使用：

- `viewModelScope` 处理页面相关任务。
- `lifecycleScope` 处理 Activity / Fragment 生命周期任务。
- WorkManager 处理可靠后台任务。

## 第三部分：取消是正常流程

用户离开页面、重复点击刷新、搜索关键字改变，都可能导致旧任务不再需要。

取消不是失败，而是告诉程序：

```text
这件事现在没必要继续做了。
```

例如搜索页面：

```text
输入 a -> 发起搜索
输入 an -> 取消上一次，搜索新关键字
输入 android -> 再取消旧任务
```

Flow 中也有很多操作符会自然处理这种场景，例如后面会学到的 `flatMapLatest`。

## 第四部分：超时

网络请求不能无限等待。

可以使用：

```kotlin
withTimeout(3_000) {
    repository.refreshLessons()
}
```

如果超时，会抛出异常。

也可以使用：

```kotlin
val result = withTimeoutOrNull(3_000) {
    repository.refreshLessons()
}
```

如果超时，返回 `null`。

这适合把“等太久”变成可控状态。

## 第五部分：并发任务的失败传播

协程中，父子任务有明确关系。默认情况下，子协程失败可能影响父协程，父协程取消也会取消子协程。

这既是保护，也是约束。

例如同时加载课程和用户信息：

```kotlin
coroutineScope {
    val lessons = async { repository.getLessons() }
    val profile = async { repository.getProfile() }

    lessons.await() to profile.await()
}
```

如果其中一个失败，整个组合任务通常也应该失败，因为页面需要的是完整数据。

## 本节小挑战

请思考三个场景：

- 用户快速连续点击刷新。
- 用户刚点刷新就退出页面。
- 网络请求超过 10 秒还没回来。

如果没有取消和超时，这三个场景会发生什么？

## 本节小结

结构化并发让异步任务不再失控。取消和超时不是边角知识，而是用户体验和工程稳定性的基本组成部分。一个成熟的 Android 工程，必须知道任务从哪里开始，也知道它应该在哪里停下。
