# 9.6 崩溃治理：Crash 收集、日志与问题分级

崩溃是最直接的稳定性问题。用户正在使用 App，突然退出，信任感会迅速下降。

崩溃治理不是只看一段堆栈，而是要能回答：发生了什么、影响多少人、为什么发生、如何修复、如何防止再次发生。

## 本节定位

本节讲崩溃治理。

重点是：

- 崩溃报告需要哪些信息。
- 可预期错误和不可预期崩溃如何区分。
- 日志如何服务定位。
- 问题如何分级。
- demo 如何模拟异常捕获。

## 学习目标

学完本节后，你应该能够：

- 理解崩溃治理需要上下文。
- 知道崩溃报告应该包含哪些信息。
- 区分可预期错误和真正崩溃。
- 设计基础日志策略。
- 写出崩溃问题分级方式。

## 第一部分：崩溃报告需要什么

一份有用的崩溃报告至少包含：

- 异常类型。
- 堆栈。
- App 版本。
- 设备型号。
- Android 系统版本。
- 发生页面。
- 用户操作路径。
- 网络和登录状态。
- 是否为 release 包。

只有堆栈还不够。上下文越完整，定位越快。

## 第二部分：可预期错误要建模

不是所有错误都应该变成崩溃。

例如：

- 网络失败。
- 数据为空。
- 登录过期。
- 课程不存在。
- 服务端返回业务错误。

这些更适合进入 UI State 或错误模型，而不是直接抛到崩溃。

第 9 章 demo 里的 `捕获一次模拟异常` 就是为了说明：可预期错误要被捕获、记录并转成可理解的报告。

## 第三部分：日志要有用

日志不是越多越好。

有用的日志应该说明：

- 哪个功能。
- 哪个动作。
- 哪个状态。
- 哪个错误码。
- 哪个关键参数。

不好的日志：

```text
走到这里了
111111
出错了
```

好的日志：

```text
CourseDetail load failed, lessonId=xxx, error=missing lessonId
```

## 第四部分：release 日志要克制

debug 包可以输出详细日志，release 包要谨慎。

release 日志不应该包含：

- token。
- 手机号。
- 邮箱。
- 身份证。
- 精确定位。
- 其他敏感业务数据。

第 9 章 demo 通过 `BuildConfig.ENABLE_VERBOSE_LOG` 展示 debug / release 日志开关的思路。

## 第五部分：问题分级

不是所有崩溃都同等优先级。

可以按影响范围和严重程度分级：

- P0：大面积崩溃、无法启动、核心路径不可用。
- P1：高频崩溃、影响主要功能。
- P2：低频崩溃、有绕过方式。
- P3：边缘场景、影响较小。

分级的目的不是推迟修复，而是让团队知道先救哪里。

## 本节小挑战

请点击 demo 中的 `捕获一次模拟异常`，写出一份崩溃报告草稿。

至少包含：

- 异常类型。
- 错误信息。
- 发生页面。
- 用户动作。
- 构建类型。

## 本节小结

崩溃治理的核心不是“看到崩溃再修”，而是建立错误建模、日志记录、问题分级和复盘流程。好的稳定性治理，会让问题更早暴露、更快定位、更少重复发生。
