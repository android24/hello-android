# 9.5 ANR 治理：耗时任务、锁等待与线程调度

ANR 是 Android 稳定性里非常重要的问题。它不像普通崩溃那样直接退出，而是让用户看到“应用无响应”的提示。

对用户来说，ANR 的体验很糟：App 像被冻住了一样。

## 本节定位

本节讲 ANR 风险和治理思路。

重点是：

- ANR 和卡顿的关系。
- 主线程为什么不能长时间阻塞。
- 锁等待、IO、广播和启动链路的风险。
- 如何在 demo 中模拟 ANR 风险。

## 学习目标

学完本节后，你应该能够：

- 理解 ANR 的基本含义。
- 知道主线程阻塞为什么危险。
- 区分卡顿和 ANR。
- 识别常见 ANR 场景。
- 写出 ANR 排查的第一步。

## 第一部分：ANR 是什么

ANR 是 Application Not Responding。

当主线程长时间无法响应系统事件时，系统可能认为应用无响应。

常见触发场景包括：

- 主线程执行耗时计算。
- 主线程等待锁。
- 主线程做磁盘 IO。
- BroadcastReceiver 执行太久。
- Service 或启动链路阻塞。

## 第二部分：卡顿和 ANR 的关系

卡顿和 ANR 都和“响应不及时”有关，但严重程度不同。

```text
短时间主线程阻塞 -> 卡顿 / 掉帧
长时间主线程阻塞 -> ANR 风险
```

第 9 章 demo 的 `模拟 ANR 风险` 只做受控短等待，不会故意制造真正长时间 ANR。它用来说明：主线程等待越久，风险越高。

## 第三部分：锁等待很隐蔽

ANR 不一定来自大循环。

有时主线程只是想拿一个锁，但锁被后台线程长时间持有。

```text
后台线程持有锁
  -> 主线程等待锁
      -> 主线程无法响应
          -> ANR 风险
```

这种问题比简单 `Thread.sleep` 更隐蔽，需要结合堆栈和 trace 分析。

## 第四部分：不要同步等待后台结果

常见危险写法是：

```text
主线程启动后台任务
主线程同步等待结果
```

这等于把异步又变成了阻塞。

更好的方式是：

- 使用挂起函数。
- 使用回调或 Flow。
- 让结果回来后更新状态。
- 不让主线程停在原地等待。

## 第五部分：ANR 排查思路

排查 ANR 可以先问：

- 主线程当时在做什么？
- 是否等待锁？
- 是否执行 IO？
- 是否在启动、广播或 Service 中阻塞？
- 是否有 Binder 调用长时间等待？
- 是否能拿到 trace 或系统日志？

后续进入 Framework 后，会继续理解系统如何发现和报告 ANR。

## 本节小结

ANR 治理的核心是保持主线程可响应。耗时任务、锁等待、IO 和同步等待都可能让主线程停下来。真正的排查要靠堆栈、trace 和上下文，而不是只看表面现象。
