# 9.2 启动优化：从点击图标到首屏展示

启动是用户和 App 的第一次见面。启动慢不一定会崩溃，但会让用户在心里先扣一分。

启动优化的第一步不是立刻删代码，而是先画出启动链路。

## 本节定位

本节讲启动性能。

目标是让你知道：

- 启动从哪里开始。
- 哪些代码会拖慢首屏。
- 如何记录关键节点。
- 为什么 Application 不应该塞太多重活。

## 学习目标

学完本节后，你应该能够：

- 理解冷启动、温启动和热启动的区别。
- 画出从点击图标到首屏展示的链路。
- 找出 Application 和首个 Activity 中的启动风险。
- 知道启动优化需要先记录时间点。
- 使用第 9 章 demo 观察启动节点。

## 第一部分：启动链路

可以先粗略理解为：

```text
点击图标
  -> 系统创建进程
      -> Application.onCreate
          -> Activity.onCreate
              -> setContent / setContentView
                  -> 首屏数据准备
                      -> 首屏展示
```

链路越长，越容易藏耗时操作。

## 第二部分：冷启动、温启动、热启动

常见分类：

- 冷启动：进程不存在，需要完整创建进程和应用。
- 温启动：进程还在，但 Activity 需要重新创建。
- 热启动：应用还在后台，回到前台较快。

课程里先重点理解冷启动，因为它最容易暴露初始化问题。

## 第三部分：Application 不要太重

很多项目启动慢，是因为 Application 里做了太多事情：

- 同步初始化多个 SDK。
- 读取大文件。
- 查询数据库。
- 预加载大量配置。
- 执行复杂计算。

Application 像是开门动作。门刚打开就搬一整套家具，用户当然要等。

更好的方式是：

- 首屏必须用的先初始化。
- 不急的延后。
- 可并行的异步化。
- 可懒加载的按需加载。

## 第四部分：首屏也要克制

首屏页面也可能拖慢启动。

常见问题：

- 首屏同步请求网络。
- 首屏一次加载大量列表。
- 首屏组合过多复杂 UI。
- 首屏等待多个模块初始化完成。

启动优化不是把所有代码从 Application 挪到 Activity，而是让首屏只做“用户马上需要看到的事情”。

## 第五部分：如何观察启动

第 9 章 demo 在 `PerformanceLabApplication` 和 `MainActivity` 中记录了一个简单启动节点：

```text
进程创建 -> 首个 Activity onCreate
```

这个数字不是完整启动指标，但它能帮助读者建立意识：

```text
先记录，再优化。
```

真实项目里可以继续使用：

- Logcat。
- Android Studio Profiler。
- Macrobenchmark。
- Baseline Profile。
- Perfetto。

## 本节实践任务

### 基础任务

- 在启动链路中增加一条 Logcat 日志。
- 记录 Application 到 Activity 的时间。
- 写下首屏不应该做的 3 件事。

### 进阶任务

- 模拟一个耗时初始化，并观察启动数字变化。
- 把耗时初始化延后到页面展示之后。
- 对比优化前后的启动记录。

## 本节小结

启动优化的核心是减少用户等待。不要凭感觉优化，要先画链路、记录节点、找到首屏真正需要的事情，再把不必要的重活延后或拆开。
