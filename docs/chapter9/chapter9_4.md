# 9.4 内存优化：泄漏、对象分配与图片资源

内存问题有时很安静。它不会立刻崩溃，而是慢慢增长，直到页面变卡、频繁 GC，甚至最终 OOM。

内存优化要关注两个问题：

```text
哪些对象太大？
哪些对象不该继续活着却还被引用？
```

## 本节定位

本节讲内存优化。

重点是：

- 对象分配。
- 引用持有。
- 内存泄漏风险。
- 图片和列表缓存。
- 如何用 demo 做内存观察。

## 学习目标

学完本节后，你应该能够：

- 理解内存增长和内存泄漏的区别。
- 知道为什么大对象和长生命周期引用危险。
- 识别 Activity / View 被错误持有的风险。
- 理解图片资源为什么容易成为内存压力来源。
- 使用 Profiler 观察内存变化。

## 第一部分：内存增长不一定是泄漏

内存增长可能是正常的。

例如：

- 用户打开了更多页面。
- 列表加载了更多数据。
- 图片缓存开始工作。

但如果页面退出后内存无法回落，就要怀疑泄漏。

所以观察内存要看趋势，而不是只看某一刻的数字。

## 第二部分：什么是泄漏

内存泄漏可以简单理解为：

```text
对象已经不需要了，
但仍然被某个地方引用，
导致它无法被回收。
```

常见风险：

- 单例持有 Activity。
- 长生命周期协程持有页面对象。
- 回调没有注销。
- 静态集合保存大对象。
- 图片缓存没有上限。

## 第三部分：第 9 章 demo 的内存实验

demo 的“内存体检”会保留 2MB 的 `ByteArray`。

连续点击 `保留 2MB`，你会看到报告变化。

点击 `释放内存` 后，demo 会清理引用并请求 GC。

这个实验的重点不是精确模拟真实泄漏，而是理解：

```text
只要引用还在，对象就不会轻易离开。
```

## 第四部分：图片资源是大头

Android 应用里的内存压力经常来自图片：

- 大图直接加载。
- 列表中重复解码。
- 没有缩略图。
- 缓存策略不合理。
- 页面退出后仍然持有图片引用。

真实项目里要优先使用成熟图片加载库，并理解缓存、采样、占位和生命周期。

## 第五部分：如何观察内存

可以用 Android Studio Profiler：

- 点击实验前记录一次内存。
- 连续点击 `保留 2MB`。
- 观察内存是否上升。
- 点击 `释放内存`。
- 观察内存是否有回落趋势。

如果要进一步分析泄漏，可以学习 LeakCanary 和 heap dump。

## 本节小结

内存优化不是只追求数字越小越好，而是确保对象生命周期合理、缓存有边界、大对象能释放。真正要警惕的是“不该活着的对象一直活着”。
