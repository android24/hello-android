# 9.7 包体积优化：资源、依赖、R8 与构建产物

包体积也是体验的一部分。包越大，下载越慢，占用越多，用户越容易在安装前放弃。

包体积优化不是最后随手压一压，而是要知道体积来自哪里。

## 本节定位

本节讲包体积和 release 构建。

重点是：

- 资源体积。
- 依赖体积。
- R8 / ProGuard。
- mapping 文件。
- 构建产物检查。

## 学习目标

学完本节后，你应该能够：

- 知道包体积常见来源。
- 理解资源和依赖为什么会膨胀。
- 知道 R8 / ProGuard 的基本作用。
- 理解 mapping 文件为什么重要。
- 为 release 构建设计体积检查。

## 第一部分：包体积来自哪里

常见来源：

- 图片资源。
- 多语言资源。
- 第三方 SDK。
- native so。
- 未使用代码。
- 重复依赖。
- 大字体、大动画、大模型文件。

优化前先看组成，不要盲目删除。

## 第二部分：资源优化

资源优化可以从这些方向开始：

- 删除未使用图片。
- 使用合适分辨率。
- 优先使用矢量图或压缩后的位图。
- 避免重复资源。
- 检查多语言资源是否必要。

资源不是越高清越好，关键是适合显示场景。

## 第三部分：依赖优化

第三方依赖很容易让包体积变大。

检查时可以问：

- 这个 SDK 是否真的必要？
- 是否引入了重复能力？
- 是否只为一个小功能引入了很重的库？
- 是否有更轻量的实现方式？
- debug 工具是否进入了 release 包？

依赖管理是工程治理的一部分。

## 第四部分：R8 / ProGuard

R8 可以帮助：

- 移除未使用代码。
- 压缩代码。
- 混淆符号。
- 优化部分字节码。

但混淆也可能带来问题，尤其是反射、序列化、第三方 SDK。

所以 release 包必须经常构建和验证。

## 第五部分：mapping 文件

开启混淆后，崩溃堆栈可能变得难以阅读。

mapping 文件用于还原堆栈。

发布时要保存：

- mapping 文件。
- native 符号表。
- 版本号和构建号。

否则线上崩溃会变得很难定位。

## 第六部分：第 9 章 demo 的体积清单

demo 中的 `quality/performance-checklist.md` 提供了基础体积检查：

- 检查未使用资源。
- 检查重复依赖。
- release 构建启用必要压缩。
- 保存 mapping 文件。

后续可以继续加入 APK Analyzer 或 Gradle 任务输出。

## 本节小挑战

请打开第 9 章 demo 的 `app/build.gradle.kts`，找到：

- debug 和 release 的区别。
- `ENABLE_VERBOSE_LOG`。
- `BUILD_PROFILE`。

说明这些配置如何服务发布检查。

## 本节小结

包体积优化要从资源、依赖、代码压缩和构建产物入手。release 构建不是最后一天才关心的事，它应该从开发阶段就持续被验证。
