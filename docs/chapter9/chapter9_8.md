# 9.8 综合实践：课程 App 性能与稳定性体检

第 9 章最后，我们把启动、卡顿、内存、ANR、崩溃和包体积放回同一个实践：给课程 App 做一次性能与稳定性体检。

这不是为了做一个完美优化报告，而是建立第一套可复用的排查路径。

## 本节定位

本节是第 9 章综合实践。

它会把前面内容合在一起：

- 记录启动节点。
- 模拟主线程卡顿。
- 把 CPU 重活放到后台。
- 观察内存分配和释放。
- 捕获一次模拟异常。
- 检查 release 构建和日志开关。
- 输出一份体检报告。

## 学习目标

学完本节后，你应该能够：

- 使用第 9 章 demo 完成一次实验。
- 为每个性能问题写观察点。
- 区分现象、原因和优化手段。
- 写出性能稳定性体检清单。
- 为后续 Framework 学习建立问题意识。

## 第一部分：体检路线

推荐路线：

```text
第 1 步：运行 demo
第 2 步：记录启动体检
第 3 步：模拟主线程卡顿
第 4 步：运行后台 CPU 实验
第 5 步：观察内存保留和释放
第 6 步：捕获一次模拟异常
第 7 步：阅读 quality 清单
第 8 步：观察性能体检分数变化
第 9 步：写体检报告
```

每一步都要写下：

- 你做了什么。
- 你观察到了什么。
- 体检分数发生了什么变化。
- 真实项目里对应什么风险。
- 你会如何治理。

## 第二部分：体检报告模板

可以这样写：

```text
问题：主线程卡顿
现象：点击按钮后页面短暂停顿
证据：主线程阻塞约 180ms
风险：用户点击延迟、动画掉帧
治理：移除主线程 sleep，把重活迁移到 Dispatchers.Default
验证：再次点击后 UI 仍可响应
```

这个模板比“优化了卡顿”更有价值，因为它保留了证据链。

## 第三部分：复盘表

| 问题 | 观察点 | 治理方向 |
| :-- | :-- | :-- |
| 启动慢 | 启动节点耗时 | 延迟初始化、减少首屏重活 |
| 滑动卡 | 主线程、重组、帧率 | 后台计算、稳定 key、减少重组 |
| 内存上涨 | Profiler、对象引用 | 释放引用、控制缓存、检查泄漏 |
| ANR | 主线程堆栈、trace | 避免锁等待、IO 和同步等待 |
| 崩溃 | 堆栈、版本、用户路径 | 错误建模、日志、问题分级 |
| 包太大 | APK Analyzer、依赖树 | 资源压缩、依赖治理、R8 |

## 第四部分：发布前体检清单

发布前至少确认：

- 启动路径没有新增明显阻塞。
- 核心页面没有明显掉帧。
- 内存没有持续异常增长。
- release 日志开关正确。
- 崩溃报告可以还原堆栈。
- 包体积变化可解释。
- 严重问题有回滚或降级方案。

这会把第 8 章的质量门禁和第 9 章的性能稳定性治理连接起来。

## 第五部分：通向 Framework

第 9 章也是进入 Framework 前的关键铺垫。

你现在已经有了这些问题：

- 启动为什么慢？
- Activity 是如何被系统启动的？
- 主线程为什么影响绘制？
- Choreographer 如何调度一帧？
- ANR 是系统如何发现的？
- 崩溃如何从线程传播到进程退出？

这些问题会自然引出后续 Framework 内容。

## 本节小挑战

请完成第 9 章 demo 的所有实验，并写一份体检报告。

至少包含：

- 一个启动观察。
- 一个卡顿观察。
- 一个内存观察。
- 一个异常观察。
- 一个体检分数变化。
- 一个 release 检查项。

## 本节实践任务

### 基础任务

- 运行第 9 章 demo。
- 点击所有实验按钮。
- 阅读 `quality` 目录下的 3 份文档。
- 写一份性能稳定性体检报告。

### 进阶任务

- 扩展每个实验的 Logcat 日志，补充更具体的场景信息。
- 改造“重置实验结果”按钮，让它同时记录一次重置日志和体检状态。
- 用 Profiler 截图记录 CPU 或 Memory 变化。
- 为课程 App 写一份发布前性能稳定性清单。

## 本章小结

第 9 章把课程 App 从“可交付”推进到“可观察、可定位、可治理”。性能优化和稳定性治理都不是玄学：先有现象，再有证据，再有改动，最后验证结果。

当你能用证据解释启动、卡顿、内存、ANR、崩溃和包体积问题时，你就已经站在了进入 Android Framework 的门口。
