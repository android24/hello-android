# 9.3 卡顿分析：主线程、掉帧与 Compose 性能

用户说“卡”，背后可能是主线程阻塞、渲染压力过大、列表重组过多，也可能是图片加载和对象创建太频繁。

卡顿治理要先找到哪一段时间被吃掉了。

## 本节定位

本节讲卡顿和渲染性能。

重点是：

- 主线程为什么不能做重活。
- 掉帧和用户感知有什么关系。
- Compose 列表为什么要关注 key 和重组范围。
- 如何用 demo 观察卡顿现象。

## 学习目标

学完本节后，你应该能够：

- 理解主线程阻塞为什么会导致卡顿。
- 知道掉帧和帧预算的基本概念。
- 识别 Compose 列表中的常见性能风险。
- 区分“CPU 重活”和“UI 绘制压力”。
- 使用第 9 章 demo 做卡顿实验。

## 第一部分：主线程负责什么

Android 主线程负责很多用户能感知的事情：

- 处理点击。
- 执行 UI 更新。
- 触发布局和绘制。
- 分发部分生命周期回调。

如果主线程被占住，用户就会感到：

- 点击没反应。
- 动画不流畅。
- 列表滑动卡。
- 页面像停了一下。

## 第二部分：掉帧是什么

屏幕刷新需要 App 在有限时间内准备好下一帧。

如果一帧准备太久，就会掉帧。

可以先这样理解：

```text
主线程太忙
  -> 下一帧准备不及时
      -> 屏幕没有拿到新画面
          -> 用户看到卡顿
```

第 9 章 demo 的 `模拟 180ms 卡顿` 会故意在主线程 `sleep`，让你感受主线程阻塞的影响。

## 第三部分：重活放到后台

CPU 密集任务不应该直接放在主线程。

例如：

```kotlin
withContext(Dispatchers.Default) {
    countPrimeNumbers()
}
```

第 9 章 demo 里的“后台 CPU 实验”会把质数计算放到 `Dispatchers.Default`，对比主线程阻塞实验，可以直观看到调度位置的差异。

## 第四部分：Compose 列表性能

Compose 列表常见风险包括：

- item 没有稳定 key。
- item 内部创建大量临时对象。
- 状态放得太高，导致大范围重组。
- 图片加载没有缓存和占位策略。
- 列表中做复杂计算。

第 9 章 demo 里使用了：

```kotlin
items(
    items = (1..40).toList(),
    key = { it }
)
```

这不是完整优化，只是提醒：列表性能要从稳定身份和重组范围开始看。

## 第五部分：观察工具

可以结合：

- Android Studio Profiler。
- Layout Inspector。
- 系统开发者选项里的帧率 / GPU 渲染分析。
- Perfetto。
- JankStats。

工具不是目的。工具的作用是让你知道“卡在哪里”。

## 本节实践任务

### 基础任务

- 解释为什么 `Thread.sleep` 不应该出现在主线程。
- 找到 demo 中后台计算的位置。
- 给 Compose 列表增加更多 item。
- 观察滚动是否仍然顺畅。

### 进阶任务

- 为列表 item 添加更复杂的内容。
- 尝试移除稳定 key，再思考可能带来的风险。
- 使用 Profiler 观察后台 CPU 实验。

## 本节小结

卡顿治理要围绕主线程、帧预算和重组范围展开。优化不是“把所有东西都异步化”，而是让 UI 线程只做它该做的事，把重活交给合适的后台执行环境。
