# 5.1 为什么需要架构

前四章让 App 逐渐具备了真实应用的样子：有页面、有交互、有跳转、有网络、有本地缓存。接下来你会遇到一个更真实的问题：代码越来越多以后，到底该放在哪里？

如果所有逻辑都堆在 Activity 或 Composable 里，一开始会很快，但很快也会变乱。页面展示、点击处理、网络请求、数据库读写、错误处理、状态转换混在一起，代码会越来越难改，也越来越难测。

第 5 章要解决的，就是 Android 工程里最经典的问题之一：**如何组织代码，让它从“能跑”走向“可维护”。**

## 本章通关画面

完成第 5 章时，你会用同一个课程列表功能，看懂几种常见架构模式的演进：

```text
Activity 里全都写
  -> MVC
  -> MVP
  -> MVVM
  -> MVI / UDF
  -> Repository / UseCase / Clean Architecture
```

这不是为了背架构名词，而是为了回答几个非常实际的问题：

- Activity 为什么会变胖？
- ViewModel 到底应该管什么？
- Repository 是不是万能杂物间？
- MVI 和 MVVM 差在哪里？
- Clean Architecture 什么时候是帮助，什么时候是过度设计？

## 本节定位

本节是 `05-android-architecture-evolution` 的开篇。

第 5 章会围绕一个统一功能展开：课程列表页面。我们会用不同架构方式组织同一个功能，让你看到每种模式解决了什么问题，又带来了什么代价。

## 学习目标

学完本节后，你应该能够：

- 理解架构不是为了复杂，而是为了控制变化。
- 说出“代码全写在页面里”的常见问题。
- 理解分层、职责边界和数据流的重要性。
- 知道 MVC、MVP、MVVM、MVI、Clean Architecture 的大致位置。
- 用“课程列表”功能作为例子观察架构演进。

## 第一部分：没有架构时会发生什么

假设一个页面要做这些事情：

- 展示课程列表。
- 点击刷新按钮。
- 请求网络数据。
- 读取本地缓存。
- 显示加载中。
- 显示错误。
- 保存用户偏好。

如果全部写在 Activity 或 Composable 里，代码很容易变成：

```text
页面代码
  + 网络请求
  + 数据库读写
  + 错误处理
  + 状态转换
  + 点击逻辑
  + 日志
```

短期看很直接，长期看很危险。

## 第二部分：架构要解决什么

架构的核心不是“套模式”，而是解决三个问题：

- 职责：每段代码负责什么？
- 边界：哪些代码不应该互相知道？
- 变化：需求变了，改动范围能不能控制住？

比如课程列表页面：

- UI 负责展示。
- ViewModel 负责页面状态。
- Repository 负责数据来源。
- UseCase 负责业务动作。
- DataSource 负责具体网络或本地读写。

当职责清晰，代码就更容易修改和测试。

## 第三部分：架构不是越复杂越好

初学架构时，很容易走向两个极端：

- 什么都不分层，所有代码堆在一起。
- 一上来就建很多层，每改一行都要穿过一堆文件。

好的架构应该和问题规模匹配。

一个只有两行文字的页面，不需要 Clean Architecture。一个有网络、缓存、权限、支付、离线和多团队协作的项目，就需要更清晰的边界。

## 第四部分：本章的学习方法

第 5 章建议用对比法学习：

```text
同一个功能
不同写法
观察职责变化
比较优缺点
总结适用场景
```

不要只问“哪个架构最好”，而要问：

- 它解决了什么问题？
- 它引入了什么成本？
- 它适合什么规模？
- 它和 Android 生命周期、Compose 状态如何配合？

## 本节小挑战

请回看前几章的示例工程，找出哪些代码属于：

- UI 展示
- 点击事件
- 状态管理
- 数据来源
- 数据转换

如果你发现某个文件同时承担了太多职责，就已经开始理解为什么需要架构了。

## 本节小结

架构的意义不是让代码看起来高级，而是让变化更可控。第 5 章会从最朴素的 MVC 开始，一步步走到 MVVM、MVI 和 Clean Architecture。
