# 7.5 模块化：从单 App 到多 Module

当代码还少时，一个 `app` 模块可以承载所有内容。但随着课程 App 继续长大，所有功能都挤在一个模块里，会逐渐出现问题：构建慢、边界乱、多人协作冲突多、业务功能互相影响。

模块化要解决的，就是工程边界问题。

这一节的画面感可以更强一点：课程 App 原本像一间大教室，所有代码都坐在里面。刚开始很热闹，后来课程列表、课程详情、同步设置、网络、数据库、设计组件都挤在一起，任何人挪一张桌子，旁边的人都要停下来让路。

## 本节定位

本节从依赖注入进入工程结构。

Hilt 解决对象如何装配，模块化解决代码如何分区。

## 学习目标

学完本节后，你应该能够：

- 理解 Android Module 的基本作用。
- 知道为什么大型项目需要模块化。
- 区分 app、feature、core、data、domain 等模块。
- 理解模块之间的依赖方向。
- 设计课程 App 的初步模块结构。

## 第一部分：为什么一个 app 模块会变重

单模块早期很舒服：

```text
app/
  ui/
  data/
  repository/
  network/
```

但随着功能增加，会出现：

- 所有代码都能互相访问。
- 修改一个小功能也可能触发大量构建。
- 不同业务团队容易改到同一片区域。
- 公共能力和业务能力混在一起。
- 测试边界不清晰。

这就像所有部门都挤在一个房间里办公，沟通很近，但迟早会吵。

当你发现“改课程详情”会影响“同步设置”，“改一个 model”会触发半个工程重编译，或者公共工具类里开始出现具体业务逻辑，模块化就不再是锦上添花，而是工程继续生长的必要空间。

## 第二部分：模块化的常见拆法

一种常见结构：

```text
app/
core/
  common/
  network/
  database/
  design/
feature/
  course/
  profile/
  settings/
data/
domain/
```

也可以按 feature 内部分层：

```text
feature-course/
  presentation/
  domain/
  data/
```

没有唯一答案，关键是边界要清楚。

## 第三部分：依赖方向

模块化最重要的是依赖方向。

一个常见方向：

```text
app -> feature -> domain
feature -> data
data -> core-network / core-database
```

需要避免：

```text
core 依赖 feature
domain 依赖 Android UI
feature 之间随意互相依赖
```

如果依赖方向乱了，模块只是文件夹变多，工程复杂度并没有降低。

## 第四部分：core 模块放什么

core 模块通常放通用能力：

- 网络基础设施。
- 数据库基础设施。
- 通用工具。
- 设计系统组件。
- 日志与监控。
- 通用资源。

core 不应该知道具体业务细节。

如果 `core-network` 里开始出现“课程列表页面逻辑”，说明边界已经漏水。

## 第五部分：feature 模块放什么

feature 模块放业务功能：

- 课程列表。
- 课程详情。
- 用户资料。
- 设置页面。

feature 可以依赖 core 能力，也可以依赖 domain 或 data 层接口。

feature 的目标是让业务功能相对独立，便于开发、测试和后续拆分。

## 本节小挑战

请把当前课程 App 设想成三个功能：

- 课程列表。
- 课程详情。
- 同步设置。

尝试为它们设计模块：

```text
feature-course-list
feature-course-detail
feature-sync-settings
```

再思考哪些代码应该放进 core。

## 本节小结

模块化的目的不是让目录变复杂，而是让边界更清晰。一个好的模块结构，应该让团队知道代码该放哪里、依赖该往哪里走、业务变化会影响哪些地方。
