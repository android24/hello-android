# 7.7 组件边界、接口设计与工程治理

依赖注入和模块化解决了很多结构问题，但它们也可能被滥用。模块越来越多，接口越来越多，Module 越写越厚，工程并不会自动变好。

真正的工程化，需要边界感。

本节像一次工程体检：不是为了把项目拆得更复杂，而是看看哪些地方已经开始疼。模块是否越来越胖？接口是否越来越难解释？一个新功能进来时，大家是否还知道它应该住在哪里？

## 本节定位

本节是第 7 章的工程治理部分。

它关注的是：当项目变大后，如何让代码仍然能被理解、被修改、被测试、被协作。

## 学习目标

学完本节后，你应该能够：

- 判断一个模块边界是否合理。
- 设计清晰的接口。
- 避免循环依赖。
- 识别过度抽象。
- 为团队协作建立基本工程规则。

## 第一部分：边界不是墙，是门

好的模块边界不是把代码隔绝起来，而是让模块通过清晰入口协作。

例如：

```text
feature-course
  -> CourseRepository 接口
      -> data-course 实现
```

feature 不需要知道网络和数据库细节，只通过接口拿课程数据。

这扇门越清楚，模块越容易替换和测试。

门的作用不是阻止协作，而是让协作有入口。入口清楚，别人就知道从哪里进来；入口混乱，最后每个人都可能从窗户翻进来，代码也会跟着变得难以维护。

## 第二部分：接口应该表达能力

好的接口表达能力，而不是暴露实现。

例如：

```kotlin
interface CourseRepository {
    fun observeCourses(): Flow<List<Course>>
    suspend fun refreshCourses()
}
```

它表达的是“观察课程”和“刷新课程”。

不好的接口可能暴露太多细节：

```kotlin
interface CourseRepository {
    fun getRetrofitApi(): CourseApi
    fun getRoomDao(): CourseDao
}
```

这等于把内部实现拆给外部看。

## 第三部分：避免循环依赖

循环依赖是模块化的大敌。

例如：

```text
feature-course -> feature-profile
feature-profile -> feature-course
```

这种结构会让修改变得危险，也容易让构建配置变复杂。

解决方式通常是：

- 抽出公共模型到 core。
- 抽出导航协议。
- 用接口隔离具体实现。
- 重新审视业务边界。

## 第四部分：警惕过度抽象

不是所有类都需要接口，不是所有功能都需要单独模块。

过度抽象的信号：

- 一个接口永远只有一个实现，也没有测试替换需求。
- 为了分层而分层，读代码要跳五六个文件。
- 模块拆得太细，但业务仍然强耦合。
- 新人不知道代码该放哪里。

工程化不是把简单问题复杂化，而是在复杂度真的出现时，用结构把它接住。

## 第五部分：团队规则

真实项目需要一些简单规则：

- 新功能放到对应 feature 模块。
- core 模块不能依赖 feature。
- domain 不依赖 Android UI。
- Repository 不直接操作 Compose。
- ViewModel 不直接创建数据源。
- 依赖方向必须可解释。

这些规则不需要一开始很完美，但要能帮助团队减少混乱。

## 第六部分：工程体检表

可以定期问：

- 哪个模块越来越胖？
- 哪些接口没人能解释？
- 哪些依赖方向很奇怪？
- 哪些对象无法测试？
- 哪些公共代码其实是业务代码？

工程治理不是一次性大扫除，而是持续体检。

## 本节小结

工程化的本质是管理复杂度。依赖注入、模块化、接口设计都只是工具。真正重要的是：边界清楚、依赖可控、变化可承受。
