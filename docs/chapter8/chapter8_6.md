# 8.6 CI、静态检查与质量门禁

当项目只有一个人开发时，你可以靠自己记得运行测试。可一旦项目进入多人协作，只靠记忆就不够了。

CI 要解决的，就是把“应该检查的事情”变成“每次都自动检查”。

## 本节定位

本节从本地测试进入团队协作。

重点不是绑定某一个 CI 平台，而是理解质量门禁的思路：

```text
代码提交
  -> 自动构建
  -> 自动测试
  -> 静态检查
  -> 产物检查
  -> 允许合并或阻止合并
```

## 学习目标

学完本节后，你应该能够：

- 理解 CI 的作用。
- 知道哪些检查适合自动化。
- 理解静态检查和测试的区别。
- 设计基础质量门禁。
- 为团队协作建立最低质量线。

## 第一部分：CI 解决什么

CI 可以帮团队发现这些问题：

- 某个分支本地能跑，别人机器不能构建。
- 提交忘了运行测试。
- 新代码引入编译错误。
- 某个模块依赖方向被破坏。
- release 变体很久没人构建，突然发版才发现失败。

CI 像一个不会偷懒的检查员。它不会替你设计架构，但能保证每次改动都经过同样的基本检查。

## 第二部分：最小 CI 检查

一开始不需要做得很复杂。

可以先做这些：

```text
./gradlew assembleDebug
./gradlew testDebugUnitTest
./gradlew lintDebug
```

如果有多环境：

```text
./gradlew assembleDevDebug
./gradlew assembleProdRelease
```

关键是先让团队形成共识：不能构建、不能通过基础测试的代码，不应该进入主分支。

## 第三部分：静态检查

静态检查不运行 App，而是检查代码本身。

它可以发现：

- 未使用资源。
- 潜在空指针风险。
- 不合理 API 使用。
- 国际化问题。
- 权限或 Manifest 配置问题。
- Compose 或 Android 常见错误。

静态检查和测试互补：

```text
测试：验证行为是否正确。
静态检查：提前发现代码和配置风险。
```

## 第四部分：质量门禁

质量门禁是一组规则。

例如：

- 编译必须通过。
- 单元测试必须通过。
- lint 不能有严重错误。
- release 变体必须能构建。
- 新增核心逻辑需要测试。
- 不能绕过代码评审直接合并。

它不是为了刁难开发者，而是为了保护主分支稳定。

主分支越稳定，团队越敢频繁交付。

## 第五部分：不要一开始门禁过重

质量门禁也要循序渐进。

如果一开始就要求：

- 所有模块 90% 覆盖率。
- 所有 UI 路径自动化。
- 所有警告零容忍。
- 每次提交跑完整端到端测试。

团队可能会被流程拖住。

更合理的方式是先建立最低线，然后逐步加严。

```text
第 1 阶段：能构建、能跑单元测试。
第 2 阶段：加入 lint 和关键 UI 测试。
第 3 阶段：加入 release 构建、覆盖率和质量报告。
```

## 本节小挑战

请为课程 App 设计第一版 CI 检查清单：

- 本地必须跑什么？
- PR 必须跑什么？
- 发版前必须跑什么？

并说明哪些检查可以以后再加。

## 本节实践任务

### 基础任务

- 写出一个最小 CI 流程。
- 标出构建、测试、lint 三个步骤。
- 给主分支定义最低质量线。

### 进阶任务

- 为 dev/prod 变体分别设计构建检查。
- 思考 UI 测试应该什么时候运行。
- 为 CI 失败设计排查顺序。

## 本节小结

CI 的价值，是把质量检查从“靠人记得”变成“系统自动执行”。它不能替代工程判断，但能让团队在同一条质量线上协作，让主分支始终保持可构建、可测试、可交付。
