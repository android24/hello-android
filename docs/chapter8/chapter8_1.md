# 8.1 为什么需要测试与质量保障

第 7 章里，我们把课程 App 整理成了更清晰的工程结构：模块有边界，对象由 Hilt 装配，环境由 Gradle 管理。现在新的问题来了：当工程继续变大，谁来保证一次修改不会悄悄弄坏已有功能？

这就是第 8 章要解决的问题。

测试不是为了让项目看起来专业，而是为了让你敢修改代码。质量保障也不是上线前才补的一张表，而是从写第一行业务逻辑时就开始积累的安全网。

## 本章通关画面

完成第 8 章后，你应该能为课程 App 设计这样一套质量链路：

```text
UseCase 单元测试
  -> Repository 测试
      -> ViewModel 状态测试
          -> Compose UI 测试
              -> Hilt 测试替换
                  -> CI 检查
                      -> 发布前质量清单
```

它不追求一开始覆盖所有代码，而是优先保护最容易出错、最值得保护的业务路径。

## 本章故事线

可以把第 8 章想象成一次课程 App 的交付前体检。

第 7 章结束时，App 的结构已经像一座有街区、有道路、有配电系统的城市。第 8 章要做的，是给这座城市装上信号灯、巡检表和报警器：

- 改 Repository 时，知道数据规则有没有坏。
- 改 ViewModel 时，知道 UI State 是否仍然正确。
- 改 Compose 页面时，知道关键按钮和文案是否还在。
- 换 Hilt 依赖时，知道测试能否使用 Fake 实现。
- 准备发布时，知道哪些检查必须先通过。

读者在这一章会从“我能写功能”走向“我能保护功能”。

## 本节定位

本节是第 8 章入口。

第 8 章主线是：

- 先理解为什么需要测试。
- 再学习单元测试如何保护业务规则。
- 然后测试协程、Flow 和 ViewModel 状态。
- 接着进入 Compose UI 测试。
- 再学习 Hilt 测试替换和测试替身。
- 最后建立 CI、质量门禁和发布前检查清单。

## 学习目标

学完本节后，你应该能够：

- 理解测试解决的核心问题。
- 区分单元测试、集成测试和 UI 测试。
- 知道哪些代码最值得优先测试。
- 理解质量保障和工程交付的关系。
- 为课程 App 设计第一版测试策略。

## 第一部分：测试到底保护什么

测试保护的不是“代码行数”，而是行为。

例如课程列表有一个规则：

```text
刷新失败时，页面应该保留旧课程，并提示错误。
```

这条规则比具体实现更重要。无论你用 Retrofit、Room、FakeDataSource，还是以后换成真实后端，只要这条规则存在，就值得被测试保护。

好的测试应该回答：

- 输入是什么？
- 触发了什么动作？
- 输出或状态应该是什么？
- 哪些副作用应该发生？

## 第二部分：没有测试时会发生什么

没有测试的项目，早期写得很快。

但当功能开始变多，就会出现熟悉的画面：

- 改一个缓存策略，课程列表突然不刷新。
- 改一个状态字段，加载中状态永远不消失。
- 改一个 Hilt Module，测试环境连到了真实接口。
- 改一个按钮文案，UI 测试找不到入口。
- 发版前只能靠手动点十几遍页面。

这就像每次出门都靠记忆检查门窗。一天两天没问题，时间久了，总会漏掉什么。

## 第三部分：测试金字塔

常见测试层次可以这样理解：

```text
UI 测试：少量，验证关键用户路径
集成测试：适量，验证多个模块协作
单元测试：大量，验证业务规则和状态转换
```

越靠下，运行越快、定位问题越容易；越靠上，越接近真实用户行为，但成本也更高。

课程里会采用务实策略：

- UseCase 和 Repository 重点写单元测试。
- ViewModel 重点测试 UI State。
- Compose UI 只保护关键路径。
- Hilt 测试重点演示如何替换依赖。

## 第四部分：先测试什么

不要一开始就追求“全覆盖”。

优先测试这些地方：

- 有明确业务规则的 UseCase。
- 负责数据组合的 Repository。
- 有加载、成功、失败状态的 ViewModel。
- 容易被改坏的关键交互。
- 发布前必须稳定的核心路径。

暂时可以不优先测试：

- 简单 data class。
- 只有一两行的纯展示 Composable。
- 没有业务规则的常量。
- 临时 Demo 代码。

测试应该先服务风险最高的地方。

## 第五部分：质量保障不只是测试

测试是质量保障的一部分，但不是全部。

一个更完整的质量体系还包括：

- 代码评审。
- 静态检查。
- 构建检查。
- 崩溃监控。
- 日志与埋点。
- 发布前清单。
- 回滚预案。

第 8 章会先把测试和发布前检查讲清楚。性能、稳定性和监控会在后续章节继续展开。

## 本节小结

测试的核心价值，是让修改变得可控。第 8 章不会把测试讲成一套冰冷流程，而是把它放回课程 App 的成长路径里：当工程越来越大，你需要一张安全网，帮你确认每次修改都没有悄悄破坏重要行为。
