# 8.8 综合实践：课程 App 质量体检

第 8 章最后，我们把测试、替身、CI 和发布检查放回同一个实践：给课程 App 做一次质量体检。

这次实践不要求你立刻写出一套大型测试体系，而是建立第一条可持续演进的质量链路。

## 本节定位

本节是第 8 章综合实践。

它会把前面内容合在一起：

- 为 UseCase 设计单元测试。
- 为 Repository 设计 Fake 数据源。
- 为 ViewModel 设计状态测试。
- 为 Compose 页面设计关键路径测试。
- 为 Hilt 依赖设计测试替换。
- 为 CI 和发布建立检查清单。

## 学习目标

学完本节后，你应该能够：

- 为课程 App 设计测试分层。
- 找出最值得优先测试的业务路径。
- 写出测试替身设计。
- 设计最小质量门禁。
- 写出发布前检查清单。
- 用一张图解释质量保障链路。

## 第一部分：目标质量链路

可以先从这条链路开始：

```text
RefreshCoursesUseCaseTest
  -> CourseRepositoryTest
      -> CourseEngineeringViewModelTest
          -> CourseEngineeringScreenTest
              -> HiltTestReplacement
                  -> CI
                      -> ReleaseChecklist
```

它不是为了追求复杂，而是覆盖从业务规则到用户路径的关键风险。

## 第二部分：优先保护课程刷新

课程刷新是一个很好的测试对象，因为它包含：

- 用户动作。
- 加载状态。
- 数据源请求。
- 缓存更新。
- 成功反馈。
- 失败处理。

可以把它拆成几层验证：

| 层级 | 关注点 |
| :-- | :-- |
| UseCase | 是否触发刷新业务动作 |
| Repository | 成功、失败、缓存规则 |
| ViewModel | `isRefreshing`、列表、提示文案 |
| UI | 用户能否点击刷新并看到结果 |
| CI | 改动后是否自动构建和测试 |

## 第三部分：测试替身设计

推荐先准备：

```text
FakeCourseRepository
FakeCourseRemoteDataSource
FakeNetworkConfig
```

它们应该能模拟：

- 正常返回课程。
- 返回空列表。
- 抛出网络错误。
- dev/prod 不同环境。
- 刷新次数变化。

这样测试不会依赖真实网络，也不会被环境状态影响。

## 第四部分：质量门禁设计

第一版质量门禁可以很简单：

```text
本地提交前：
  - 编译当前模块
  - 运行相关单元测试

PR 合并前：
  - assembleDebug
  - testDebugUnitTest
  - lintDebug

发布前：
  - assembleProdRelease
  - 关键路径手动验收
  - 发布检查清单确认
```

后续再逐步加入 UI 测试、覆盖率、截图测试和性能检查。

## 第五部分：课程 App 质量说明书

完成本章后，你可以为课程 App 写一份质量说明书。

至少包含：

- 哪些模块有单元测试。
- 哪些业务规则被测试保护。
- 哪些依赖在测试中被替换。
- 哪些检查由 CI 自动执行。
- 哪些检查必须发布前人工确认。
- 如果发布失败，如何回滚或止损。

这份说明书会让项目不只是“有代码”，而是“有质量策略”。

## 第六部分：复盘表

| 问题 | 质量手段 |
| :-- | :-- |
| 改业务规则怕影响旧逻辑 | UseCase / Repository 单元测试 |
| ViewModel 状态变化容易漏 | ViewModel 状态测试 |
| 页面关键按钮可能消失 | Compose UI 测试 |
| 测试依赖真实网络 | Fake / Hilt 测试替换 |
| 提交前忘记跑测试 | CI |
| release 包配置不确定 | 发布前检查清单 |
| 主分支经常被破坏 | 质量门禁 |

## 本节小挑战

请为第 7 章示例工程写一份“质量体检报告”，包括：

- 最值得测试的 3 条业务规则。
- 需要准备的 3 个 Fake。
- 一条 ViewModel 状态测试路径。
- 一条 Compose UI 测试路径。
- 一份最小 CI 检查清单。
- 一份发布前检查清单。

如果这份报告写清楚，第 8 章就真正落地了。

## 本节实践任务

### 基础任务

- 为课程刷新设计测试分层。
- 写出 Fake Repository 的能力清单。
- 为 ViewModel 写状态转换表。
- 写出最小 CI 流程。

### 进阶任务

- 给第 7 章示例工程补第一批测试。
- 为 Hilt 测试替换设计 Module。
- 为 `prodRelease` 写发布检查表。
- 写一份质量说明书放到示例工程 README 中。

## 本章小结

第 8 章把课程 App 从“工程结构清晰”推进到“质量路径清晰”。测试让修改更放心，Fake 让测试更可控，CI 让检查更稳定，发布清单让交付更可靠。

当你能说清楚一个功能如何被测试、一次提交如何被检查、一个版本如何被交付时，你就开始具备真实 Android 工程里的质量意识。
