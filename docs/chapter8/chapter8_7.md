# 8.7 发布前检查：从能运行到可交付

一个 App 能在开发机上运行，不代表它已经可以交付。

真正发布前，需要确认的不只是“页面能打开”，还包括构建变体、签名配置、权限、隐私、崩溃风险、日志开关和回滚方案。

第 8 章到这里，开始从测试走向交付。

## 本节定位

本节讲发布前检查。

它不是完整发布体系的全部内容，而是帮助你建立一个意识：

```text
交付不是点击 Build 按钮。
交付是一组可复查、可追踪、可回滚的工程动作。
```

## 学习目标

学完本节后，你应该能够：

- 理解开发包、测试包和正式包的区别。
- 知道发布前需要检查哪些配置。
- 设计基础发布清单。
- 理解日志、接口、权限和签名的风险。
- 为课程 App 建立交付意识。

## 第一部分：发布前最怕什么

发布前最怕的是“以为”。

- 以为是正式接口，其实还是测试接口。
- 以为日志关了，其实 release 还在输出调试信息。
- 以为混淆没问题，其实关键类被错误处理。
- 以为权限没变，其实新增了敏感权限。
- 以为包名正确，其实装到了测试包环境。

发布检查清单就是为了减少这些“以为”。

## 第二部分：构建变体检查

第 7 章讲过 buildTypes 和 productFlavors。

发布前至少要确认：

- 当前构建的是不是 `prodRelease`。
- `applicationId` 是否正确。
- `versionCode` 是否递增。
- `versionName` 是否符合发布计划。
- `BASE_URL` 是否指向正式环境。
- 调试开关是否关闭。

这些配置最好不要靠人肉记忆，而应该尽量由 Gradle 管理。

## 第三部分：签名与混淆

正式发布通常需要签名。

还可能涉及：

- R8 / ProGuard。
- 资源压缩。
- mapping 文件保存。
- native 符号表保存。
- 第三方 SDK 混淆规则。

这部分一旦出问题，可能表现为：

- release 包崩溃，debug 包正常。
- 某些反射代码找不到类。
- 崩溃堆栈无法还原。

所以 release 变体必须经常构建，不应该等发版当天才第一次试。

## 第四部分：权限与隐私

发布前要检查权限。

例如：

- 是否新增了定位、相机、麦克风、存储等敏感权限。
- 权限用途是否能解释清楚。
- 隐私政策是否需要更新。
- 用户是否能拒绝权限后继续使用基础功能。

权限不是 Manifest 里的一行配置，它会影响用户信任。

## 第五部分：日志与调试开关

正式包不应该暴露过多调试信息。

需要检查：

- 是否关闭详细日志。
- 是否隐藏测试入口。
- 是否关闭调试面板。
- 是否禁止连接测试接口。
- 是否避免输出 token、手机号、邮箱等敏感信息。

日志是排查问题的朋友，但在正式环境里也可能变成风险。

## 第六部分：回滚与观察

发布不是结束，而是开始观察。

发布后至少要关注：

- 崩溃率。
- 启动失败。
- 接口错误率。
- 核心路径转化。
- 用户反馈。

如果出现严重问题，要知道：

- 是否能灰度暂停。
- 是否能回滚版本。
- 是否能关闭某个开关。
- 谁负责响应。

这些会在后续稳定性治理章节继续展开。

## 本节小挑战

请为课程 App 写一份 `prodRelease` 发布前检查清单，至少包含：

- 环境。
- 版本号。
- 签名。
- 日志。
- 权限。
- 测试结果。
- 回滚预案。

## 本节实践任务

### 基础任务

- 列出 debug 包和 release 包的差异。
- 检查 dev/prod 环境配置。
- 写一份发布前检查表。

### 进阶任务

- 思考哪些检查可以放进 CI。
- 思考哪些检查必须人工确认。
- 为一次失败发布写出回滚流程。

## 本节小结

可交付不是“能跑一下”，而是经过检查、能被追踪、出了问题能处理。发布前检查清单会让工程从开发状态进入交付状态，也会让你开始具备真实项目里的质量意识。
