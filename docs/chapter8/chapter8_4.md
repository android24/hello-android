# 8.4 Compose UI 测试：验证关键用户路径

单元测试能保护业务规则，ViewModel 测试能保护状态变化。但有些问题只有到了 UI 层才会暴露：按钮是否还在、文案是否能找到、点击后页面是否变化。

这就是 Compose UI 测试的价值。

## 本节定位

本节讲 Compose UI 测试。

它不会把 UI 测试写成“把页面每个像素都检查一遍”，而是聚焦关键用户路径：

```text
用户看到了什么？
用户点击了什么？
页面是否给出正确反馈？
```

## 学习目标

学完本节后，你应该能够：

- 理解 UI 测试适合保护什么。
- 使用语义节点思考 Compose 页面。
- 知道为什么不应该过度依赖视觉细节。
- 为关键按钮和状态文案设计测试。
- 区分 UI 测试和 ViewModel 测试的职责。

## 第一部分：UI 测试适合保护什么

UI 测试适合保护用户路径。

例如：

```text
打开课程工程化页面
  -> 看到当前环境
  -> 点击刷新按钮
  -> 看到刷新结果
```

它不适合测试每一个圆角、颜色、间距。那些视觉细节更适合通过设计规范、截图检查或人工验收来把关。

UI 测试最重要的是确认用户还能完成关键动作。

## 第二部分：Compose 的语义树

Compose UI 测试通常不是直接找某个 View ID，而是通过语义信息查找节点。

例如：

```text
找到文本“通过 Hilt 刷新课程”
点击它
断言出现“刷新完成”
```

这也提醒我们：页面上的关键操作和关键状态，应该有稳定、可理解的文本或语义描述。

如果一个按钮对用户重要，它也应该对测试友好。

## 第三部分：不要把 UI 测试写得太脆

脆弱的 UI 测试经常因为小改动失败：

- 文案改了一个字就失败。
- 列表顺序稍微变化就失败。
- 颜色和间距变化就失败。
- 动画时机导致偶发失败。

更稳的策略是：

- 只测试关键路径。
- 使用稳定的测试标签或语义描述。
- 避免断言过多视觉细节。
- 让测试数据可控。

测试应该帮你发现问题，而不是每天制造噪音。

## 第四部分：和 ViewModel 测试分工

同一个功能可以分层测试。

ViewModel 测试：

```text
点击刷新动作触发后，UI State 是否更新？
```

UI 测试：

```text
用户点击刷新按钮后，页面是否显示刷新结果？
```

前者保护状态逻辑，后者保护用户路径。两者不是互相替代，而是互相补位。

## 第五部分：测试入口要清楚

为了让 Compose UI 容易测试，页面最好拆成两层：

```text
Route：连接 ViewModel
Screen：接收 uiState 和事件回调
```

例如：

```kotlin
@Composable
fun CourseScreen(
    uiState: CourseUiState,
    onRefresh: () -> Unit
)
```

这样测试 `CourseScreen` 时，可以直接传入假状态和假回调，不一定需要启动完整 Hilt 环境。

这就是可测试性反过来推动 UI 结构变清晰。

## 本节实践任务

### 基础任务

- 为课程列表页面设计一条 UI 测试路径。
- 标出需要查找的文本或测试标签。

### 进阶任务

- 将 Route 和 Screen 分离。
- 为 Screen 准备一个假 UI State。
- 为刷新按钮设计点击测试。
- 思考哪些视觉细节不应该进入自动化 UI 测试。

## 本节小结

Compose UI 测试的目标，是保护关键用户路径。它不应该替代单元测试，也不应该变成像素级审查。好的 UI 测试像一次稳定的人工验收：打开页面，完成动作，确认结果。
